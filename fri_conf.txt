Here’s the diff between Phase 1 and Phase 2 of the platform:

What Changed: Phase 1 → Phase 2
Storage & Infrastructure
	•	Phase 1 used Dataverse for archival; Phase 2 moved to Azure Blob Storage (file storage) + Azure PostgreSQL (metadata), giving cheaper storage, faster I/O, and proper SQL querying for audit.
Volume Handling
	•	Phase 1 generated a single Excel per export, which broke when datasets crossed Excel’s ~10-lakh row limit. Phase 2 auto-splits large exports into numbered parts (_part1.xlsx, _part2.xlsx, …), each uploaded and archived independently.
Memory Footprint
	•	Phase 1 built Excel files in standard mode (full workbook held in RAM). Phase 2 uses openpyxl write-only streaming, so memory stays flat regardless of row count — large exports no longer crash the App Service worker.
Download Mechanism
	•	Phase 1 streamed the file straight to the user’s browser at generation time. Phase 2 uploads to Blob first, then serves short-lived SAS URLs — making files re-downloadable any time without re-running the export.
Centralized Repository
	•	Phase 1 had no persistent central store beyond Dataverse rows. Phase 2 introduces a true repository: every file lives at a predictable blob path under a YYYY-MM-DD/ folder, with a corresponding export_files row carrying module, blob path, record count, date, and timestamp.
New Capability — Matchoff (Phase 2 only)
	•	Phase 1 had no mobile-number matching; downstream teams did this manually in Excel. Phase 2 adds a full Matchoff workflow: pick any archived file, match its mobile numbers against the 30-lakh mobilecleanup table via parallel chunked SQL (1,000 per chunk, 5 workers, pooled connections), and download an enriched Excel with cust_id attached.
Reference Data Store (Phase 2 only)
	•	Phase 2 introduces the mobilecleanup PostgreSQL table — a one-time bulk load of the Bank’s ~30 lakh internal customer mobile numbers, indexed on final_mob_num — to power Matchoff lookups.
Audit & Compliance
	•	Phase 1 logged exports loosely in Dataverse. Phase 2 has dedicated audit tables: export_files for every export (and every part of a split export) and matchoff_runs for every matching job, with source file, input rows, matched count, and run date — giving a fully reconstructable RBI audit trail.
Performance Engineering
	•	Phase 2 adds psycopg2 connection pooling (ThreadedConnectionPool) and chunked blob uploads for files > 64 MB, so parallel workers reuse connections and large uploads don’t spike memory.
Turnaround
	•	Phase 1: ~35–40 seconds for a standard export, then manual matching in Excel afterwards (hours).
	•	Phase 2: ~35–40 seconds for export, plus seconds-to-minutes for matchoff on the same UI — end-to-end under a minute for most requests.​​​​​​​​​​​​​​​​
